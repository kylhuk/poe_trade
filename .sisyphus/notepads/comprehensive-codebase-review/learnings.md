- Scanner runtime identity is currently split: `item_or_market_key` is a hash of source row JSON (`cityHash64`), while `alert_id` is run-scoped (`scanner_run_id + item_or_market_key`), so alert IDs are not stable opportunity IDs across reruns.

- `search_hint` exists today only as a CLI best-effort derived display field (`poe_trade/cli.py::_best_effort_search_hint`) and is not persisted in `scanner_recommendations` or exposed in `/api/v1/ops/scanner/recommendations`.
- Frontend has no dedicated opportunities tab (`frontend/src/pages/Index.tsx` tabs: dashboard/services/analytics/pricecheck/stash/messages), and analytics diagnostics/session panels are explicit `feature_unavailable` placeholders.
- Journal and ML surfaces exist but are disconnected from the scanner opportunity contract in API/UI: journal is CLI + DB table backed (`journal_events`, `journal_positions`) with no `/api/v1/journal/*` routes, and ML currently feeds analytics/price-check rather than scanner recommendation payloads.
